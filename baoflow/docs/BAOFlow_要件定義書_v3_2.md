## 1. プロジェクト概要

### 1.1 事業概要
日本のクライアント（カフェ、食品メーカー等）からパッケージ製造の要望を受け、中国の工場で製造し、日本へ輸入・納品する仲介事業。製造・輸入に加え、提携ロジセンターでの在庫保管・受発注代行サービスも提供。現在、見積もり取得・原価計算・クライアント提案・発注・配送手配・在庫管理を手作業で行っており、これを一元管理するWebサービスを構築する。

### 1.2 現状の課題
- **見積もりの往復が煩雑**: 1案件あたり平均2〜3回の見積もり修正、さらに数量別の複数パターン見積もりが発生
- **原価計算が手作業**: 工場単価×数量＋送料＋版代→掛率適用→円換算→税込計算を毎回手計算
- **コミュニケーションが分散**: LINE、Email、Messenger、Slack、WeChat、Lark、Alibabaと多チャネル
- **データ管理が困難**: Larkの表計算で管理しているが、見積もりバージョン管理・転記・日本語変換に時間がかかる
- **スケール限界**: 現状の仕組みでは1人あたり月10件が限界
- **クライアントの曖昧な要望**: クライアントは自分が何を欲しいか明確にわかっていないことがほとんど
- **工場とのやり取り回数**: 送料計算に住所が必要、配送方法の確認、仕様の曖昧さによる再確認など、往復が多すぎる
- **デザインデータの散逸**: AIで作ったテンプレートや修正版がチャットやメールに散らばり、最終版がどれかわからなくなる
- **在庫保管業務が煩雑**: 提携ロジセンターへの事前通告、在庫証明の発行、月次保管料の請求を全て手作業で処理

### 1.3 設計思想：工数削減ファースト
本システムの最重要設計方針は **クライアント・工場双方とのやり取り回数の最小化** である。やり取り1回の削減が、営業1人あたりの処理可能案件数に直結する。

具体的なアプローチ:
- クライアントからは「曖昧でもいいから一度に全部もらう」（複数案件の一括依頼、柔軟な入力フォーム）
- 工場には「一発で全パターン出してもらう」（配送方法・Incoterms・日数の複数同時依頼）
- 定型メッセージとAIアシスタントで人手の介在を減らす（完全自動モード対応）
- プログレスバーでステータスを自動共有し、「今どうなっている？」の問い合わせを排除
- すべてのコミュニケーション（クライアント・工場・社内）をシステム内チャットに集約
- デザインデータ・帳票・チャット・進捗を案件に紐づけ一元管理

### 1.4 目標
- 営業1人あたり月20〜30件をこなせる体制の構築
- 2〜3年で営業チームを10〜20名規模に拡大
- 見積もり作成〜クライアント提案までの時間を50%以上短縮
- クライアント・工場との平均やり取り回数を50%削減
- 在庫保管サービスの事務作業（通告・在庫証明・請求）を90%自動化

## 2. ユーザーロールと利用シナリオ

### 2.1 ユーザーロール

| ロール | 説明 | 人数（現在→目標） |
|--------|------|-------------------|
| **管理者（Admin）** | システム全体の管理、マスター設定、ダッシュボード閲覧 | 1名 |
| **営業担当者** | 案件の作成・見積もり作成・クライアント対応・発注管理 | 2名 → 10〜20名 |
| **クライアント（日本企業）** | 発注依頼・進捗確認・見積もり承認・リピート発注 | 10社 → 50社以上 |
| **工場担当者（中国工場）** | 見積もり回答・仕様確認・製造進捗報告 | 10工場 → 拡大 |

### 2.2 主要利用シナリオ

**シナリオA: 新規案件（一括見積もり依頼〜成約）**
1. クライアントが複数商品の要望を一括入力（曖昧な記述もOK、既存見積書の添付も可）
2. 営業がシステム上で要望を確認、必要に応じてチャットで補足
3. 営業が複数案件を複数工場に振り分けて見積もり依頼（相見積もり対応）
4. 工場がExcelフォーム or システムフォームで一括回答（複数配送方法・日数含む）
5. システムが工場Excelを自動パースし、原価計算→販売価格を自動算出
6. 営業が確認・調整後、クライアントにシステム上で見積書を提示
7. クライアントがプログレスバー上で見積もり確認→承認 or 修正依頼
8. 承認 → 発注確定 → 請求書自動発行

**シナリオB: 製造〜納品（工場への支払いフロー含む）**
1. 発注確定 → 請求書発行 → クライアントからの入金確認
2. **工場への前払い（半金 or 全額）** → 支払い方法: Wise / Alibaba（クレジットカード）
3. 工場が前払い入金を確認 → 製造開始
4. 工場が製造進捗を更新 → プログレスバーに反映
5. 完成 → パッキングリスト作成（自動採番・段ボール明細）
6. **工場への残金支払い**（半金の場合）
7. 発送 → トラッキング番号をシステムに登録
8. システムがトラッキング情報を自動取得・表示
9. 到着 → 検品（直接納品 or ロジセンター納品を選択）
10. 納品完了 → 納品書自動発行
11. 各ステータス変更時: クライアントにメール通知、営業・工場にチャット通知

**シナリオC: リピート注文**
1. クライアントが過去の注文履歴から「再注文」ボタンを押下
2. 前回と同じ仕様で数量のみ変更可能
3. 最新の工場単価・為替レートで自動再計算
4. 営業確認 → 発注

**シナリオD: 在庫保管サービス**
1. 案件の納品先を「提携ロジセンター」に設定
2. 出荷前にシステムが自動でロジセンターに事前通告メール送信（商品名・数量・到着予定日・カートン数）
3. ロジセンターに到着 → 入庫確認 → 在庫数に反映
4. クライアントがBAO Flowポータルから出庫依頼 → 営業承認 → ロジセンターに出庫指示メール → 出荷
5. 月末に自動で在庫証明書PDF発行 → クライアントに送信
6. 月末に保管料を自動計算 → 保管料請求書を自動発行

## 3. 機能要件

### 3.1 マスターステータスフローとロール別プログレスバー

**設計思想: 案件のライフサイクルは一本の長いフローとして内部管理する。ただし各ロール（クライアント・営業・工場）は関心事が異なるため、同じマスターフローから派生した専用のプログレスバーを表示する。**

**マスターステータスフロー（内部管理用・全25ステップ）:**

```
【見積もりフェーズ】
M01: 見積もり依頼受付
M02: 営業確認・工場選定中
M03: 工場見積もり依頼送信済み
M04: 工場回答待ち
M05: 工場回答受領・原価計算完了
M06: クライアントへ見積もり提示
M07: クライアント検討中
M08: クライアント修正依頼
M09: 見積もり再調整中
M10: 見積もり再提示
M11: クライアント承認

【発注・支払いフェーズ】
M12: 請求書発行
M13: クライアント入金待ち
M14: クライアント入金確認
M15: 工場へ前払い（半金 or 全額）
M16: 工場入金確認・製造開始待ち

【製造フェーズ】
M17: 製造開始
M18: 製造中（進捗%）
M19: 製造完了・検品

【出荷・納品フェーズ】
M20: 工場残金支払い（該当する場合）
M21: 発送準備・パッキングリスト作成
M22: 発送済み（トラッキング番号発行）
M23: 輸送中
M24: 到着・検品（直接納品 or ロジセンター入庫）
M25: 納品完了
```

**クライアント向けプログレスバー（7ステップに集約）:**

```
[依頼受付] → [見積もり確認] → [承認・お支払い] → [製造中] → [発送済み] → [到着] → [納品完了]
  M01        M06-M10          M11-M14          M17-M19    M22        M24       M25
```

| ステップ | 表示テキスト | アクション |
|---------|------------|-----------|
| 依頼受付 | 「ご依頼を受け付けました。お見積もりを準備中です。」 | — |
| 見積もり確認 | 「お見積書をご確認ください。」 | [見積書DL] [承認] [修正依頼] |
| 承認・お支払い | 「ご承認ありがとうございます。請求書をご確認ください。」 | [請求書DL] |
| 製造中 | 「製造を開始しました。（進捗: XX%）」 | — |
| 発送済み | 「発送されました。」 | [トラッキング確認] |
| 到着 | 「到着しました。検品中です。」 | — |
| 納品完了 | 「納品完了しました。」 | [納品書DL] [リピート注文] |

**営業向けプログレスバー（全ステップ表示 + アクション指示）:**
- マスターステータス全25ステップを詳細表示
- 各ステップで「次に何をすべきか」をアクションボタンで明示
- 支払い関連ステータス（M13-M16, M20）にはアラート色で表示
- 停滞アラートとの連動

**工場向けプログレスバー（6ステップに集約）:**

```
[見積もり依頼] → [見積もり回答済] → [入金確認・製造開始] → [製造中] → [発送準備] → [発送完了]
    M03-M04          M05             M16                   M17-M19    M21         M22
```

| ステップ | 表示テキスト | アクション |
|---------|------------|-----------|
| 見積もり依頼 | 「新しい見積もり依頼があります」 | [回答する] |
| 見積もり回答済 | 「回答済み。選定結果をお待ちください」 | — |
| 入金確認・製造開始 | 「入金が確認されました。製造を開始してください」 | [製造開始を記録] |
| 製造中 | 「製造中です」 | [進捗%更新] |
| 発送準備 | 「製造完了。パッキングリストを提出してください」 | [パッキングリスト提出] |
| 発送完了 | 「発送情報を入力してください」 | [トラッキング番号入力] |

**通知ルール:**
- ステータス変更時 → クライアントにはメール通知（システム内チャットにも反映）
- ステータス変更時 → 営業にはシステム内チャットに自動メッセージ
- ステータス変更時 → 工場にはシステム内チャットに自動メッセージ（中国語）
- 支払い関連（M13, M15, M20）→ 営業に要対応アラート

### 3.3 原価計算エンジン（自動化の核心）

**計算ロジック（サンプル代を原価から除外）:**

```
Step 1: ドル合計金額 = (工場単価 × 数量) + 送料合計 + 版代 + その他費用
         ※サンプル代は含めない（別途管理）
         ※食品検査費用がある場合は加算（下記参照）

Step 2: 商品単価（ドル） = ドル合計金額 ÷ 数量

Step 3: 販売単価（ドル） = 商品単価 ÷ 掛率
         ※掛率 = 原価率（例: 0.65なら原価が売値の65%）

Step 4: 販売単価（円） = 販売単価（ドル） × 為替レート → 端数調整

Step 5: 請求合計（税抜） = 販売単価（円） × 数量
         ※サンプル費用（クライアント負担分累計）がある場合は加算

Step 6: 請求合計（税込） = 請求合計（税抜） × 1.1
```

**手数料の自動計算（調査結果に基づく）:**

| 支払い方法 | 手数料体系 | デフォルト設定 |
|-----------|-----------|-------------|
| **Wise送金** | 通貨変換手数料（変動%） + Swift固定費 | 変換: 約0.6%、Swift: $7.41〜$15.47/件 |
| **Alibaba CC** | 取引額の一律% | **2.99%** |
| **銀行振込(T/T)** | 固定手数料 | 約$40/件（銀行により異なる） |

- Wise手数料の詳細: JPY → USD変換時に約0.35〜0.85%の変換手数料 + USD → 中国宛Swift送金時に$7.41（コルレス手数料なし）〜$15.47（コルレス手数料込み）。月間$25,000 USD相当以上の送金で割引適用。
- Alibaba CC手数料の詳細: クレジットカード/デビットカード/PayPal/Apple Pay/Google Pay共通で取引額の**2.99%**。T/T（電信送金）は約$40/件の固定手数料。
- 手数料率は**設定画面で随時変更可能**（Wiseの料率改定等に対応）

**食品検査費用（食品衛生法に基づく輸入検査）:**
- 食品に直接触れる包装容器（カップ・蓋・バッグ等）は食品衛生法の対象
- 検査種類と費用目安:
  - 届出審査: **無料**（検疫所による書類審査）
  - 自主検査（指導検査）: **¥15,000〜¥50,000/品目**（材質・検査項目により変動）
  - 申請代行料（通関業者経由）: **¥5,000〜¥10,000/件**
  - 命令検査: 品目により大幅に変動（稀なケース）
  - モニタリング検査: **無料**（国費負担）
- 品目登録制度により、同一製品・同一工場の2回目以降は検査省略可能
- 検査費用は設定画面で品目別に管理可能

**掛率のスライダー調整:** 営業が掛率を自由に微調整→販売単価がリアルタイムで再計算

**配送オプション比較:** 同じ商品に対し、船便EXW / 船便DDP / 空輸EXW / 空輸DDP の各合計を一覧比較

**複数数量パターン:** 例: 5,000個/10,000個/20,000個の3パターンを同時計算

### 3.4 帳票自動生成（見積書・請求書・納品書）— Misocaから完全移行

**共通仕様:**

- 各帳票の自動採番（見積書: Q-YYYYMMDD-NNN、請求書: INV-YYYYMMDD-NNN、納品書: DL-YYYYMMDD-NNN）

- プログレスバーの該当ステータスからワンクリックでダウンロード可能
- 振込先情報（銀行名・支店名・口座種別・口座番号・名義）を設定画面で追加・編集・削除可能（複数口座対応）

**② 請求書（Invoice）— 現Misocaフォーマットを踏襲**
- 発行タイミング: マスターステータス M12「請求書発行」時
- レイアウト（現行Misocaフォーマットベース）:
  - ヘッダー: 日付、請求番号、自社情報（社名・住所・TEL・メール）、角印、**適格請求書番号（インボイス番号）**
  - 宛先: 「{クライアント名} 様」
  - 請求金額（税込合計を大きく表示）
  - お支払い期限
  - 明細テーブル: 品番・品名 / 数量 / 単価 / 金額

    - （例:「サンプル製作費用【60,060円(送料込) 弊社半額負担】」）
    - ※サンプル費用はクライアント負担分のみ記載
  - 小計 / 消費税(10%) / 合計
  - 備考欄: リードタイム、春節等の休業案内、その他注意事項（テンプレート+案件個別入力）
  - 振込先情報

**③ 納品書（Delivery Note）**
- 発行タイミング: マスターステータス M25「納品完了」時
- 内容: 納品番号、商品明細、数量、納品日、受領確認欄

### 3.10 チャット・コミュニケーション

**設計思想: 全てのコミュニケーションをシステム内に集約。1つのツールで全て管理できる状態を作る。**

**チャットの種類（案件ごとに自動作成）:**
1. **クライアント ↔ 営業チャット**: クライアントからの質問・要望、営業からの提案・確認
2. **営業 ↔ 工場チャット**: 見積もり依頼、仕様確認、進捗確認（日中同時翻訳付き）
3. **社内チャット**（将来）: 営業同士・営業-管理者のやりとり

**将来構想: 工場専用アプリ**
- 全機能が完成した後の次のステップとして、工場側専用のモバイルアプリを開発予定
- 工場担当者がアプリ上で直接やりとり・ステータス更新・見積もり回答を行える
- それまではWeb版ポータル + メールブリッジで対応

**メールブリッジ機能（核心機能）:**
- チャットでメッセージ送信 → 相手にメール通知が飛ぶ（システムメールから送信）
- 相手がメールで返信 → チャットにメッセージとして反映される
- これにより、クライアントはシステムにログインしなくてもメール返信でコミュニケーション可能
- 工場も同様（WeChat等にログインせずメール返信で対応可能）
- メールのSubjectに案件IDを含め、正しいチャットスレッドに自動振り分け

**日中同時翻訳（営業 ↔ 工場チャット）:**
- 営業（日本語）↔ 工場（中国語）の同時翻訳
- 英語が母国語でない双方にとって、直接の日中翻訳の方が精度が高い
- 元のメッセージも表示（翻訳精度の確認用）
- 技術用語の辞書登録機能（パッケージ業界の専門用語）

**定型メッセージ機能:**
- よく使うメッセージをテンプレートとして登録
- ボタン1つで送信可能
- 自動翻訳済みの定型文ペア（日本語↔中国語）
- カテゴリ分け: 見積もり依頼 / 仕様確認 / 進捗確認 / 督促 / 承認・感謝

**ファイル共有:**
- チャット内で画像・ファイル添付対応
- 共有されたファイルは案件のファイル一覧にも自動登録
- デザインデータのやり取りもチャット経由で可能

### 3.11 AIアシスタント機能

**2つの動作モード:**

**モード1: アシストモード（デフォルト）**
- AIがメッセージの下書きを作成 → 営業がワンクリックで送信 or 編集
- 工場からの回答を解析し、必要情報が揃っているか自動チェック（不足があれば再質問を提案）
- クライアントへの見積もり説明メールの自動下書き

**モード2: 自動モード（営業が案件単位でON/OFF）**
- 定型的なやり取り（見積もり依頼送信、進捗確認、入金確認後の製造開始通知等）をAIが承認なしで自動実行
- **判断に迷う場合のみ営業に指示を仰ぐ**（例: 工場の回答に矛盾がある、クライアントの修正依頼が曖昧、価格が前回から大幅に変動）
- 自動実行したアクションはログに記録 + 営業に通知（まとめて確認可能）
- 営業はいつでもモードを切り替え可能

**自動処理の対象タスク:**
| タスク | アシストモード | 自動モード |
|--------|-------------|-----------|
| 工場への見積もり依頼送信 | 下書き作成 | 自動送信 |
| 工場への進捗確認 | 下書き作成 | 自動送信（週1回） |
| 工場回答の不足チェック | 不足項目を提示 | 自動で再質問 |

### 3.17 商品カタログ（クライアント向け参考資料）

**設計思想: クライアントは自分が何を欲しいか明確にわかっていないことがほとんど。「まず実物を見たい」という要望に応えるため、カタログを通じて「こんな感じのものが欲しい」というイメージを持ってもらう。**

**重要: カタログはどの工場のどの商品かを見せるものではなく、「当社で扱える商品ラインナップ」として見せる。工場の選定は営業側で管理する。**

**カタログ構成:**
- 商品カテゴリ別に閲覧可能
- 各商品に: 写真、タイプ名（日英）、サイズ一覧、カラー一覧、素材（分かりやすい表記）、オプション、カスタム印刷可否、MOQ目安、参考価格帯
- 「この商品で見積もり依頼」ボタン → 見積もり依頼フォームに仕様が自動入力

**管理画面（営業・管理者向け）:**
- カタログアイテムの追加・編集・削除
- 表示/非表示の切り替え・並び替え・おすすめフラグ
- 内部メモ: 推奨工場・原価目安等（クライアント非表示）

**AIカタログ一括取込機能（Smart Catalog Import）:**

工場カタログPDFや画像を一括アップロードすると、AIが自動で商品情報を抽出・構造化する機能。

- **入力:** 工場カタログPDF / 画像（複数ページ対応）
- **AI処理:**
  1. OCR + 画像認識で各商品を検出
  2. 商品名・サイズ・素材・カラー・価格・MOQ等を自動抽出
  3. 商品画像を自動切り抜き
  4. BAO Flowのカタログデータ構造にマッピング
- **出力:** 仮登録状態のカタログアイテム一覧（営業が確認・修正後に公開）
- **営業の作業:** AI抽出結果を確認 → 誤りを修正 → 不要な商品を除外 → 工場情報を内部メモに紐付け → 公開
- **対応フォーマット:** PDF / JPG / PNG / Excel（工場の見積もり表も対応）
- **注意:** 工場名や原価はクライアント向けカタログからは自動除去

### 3.19 在庫保管サービス管理（CO-NECT完全廃止 → BAO Flow内で受発注機能を実装）

**事業概要:**
クライアントの中には、完成品を一括で受け取る場所がなく「預かって欲しい」という要望が多い。提携ロジセンターに直納品し、月額保管料を徴収。現在CO-NECTで行っている受発注機能もBAO Flow内に統合し、外部サービスへの依存を完全に排除する。

**現状の課題:**
- ロジセンターへの事前通告を毎回手動で行っている
- 在庫証明の発行依頼が来るたびに手作業
- 月末の保管料計算・請求も手作業
- CO-NECTという別サービスで受発注管理 → BAO Flowと二重管理になっている

**システムで自動化する範囲:**

**① ロジセンター入庫管理:**
- 案件の「納品先タイプ」が「ロジセンター」の場合に自動発動
- 発送済み（M22）になった時点で、ロジセンターに自動で入庫通告メールを送信
  - メール内容: 案件ID、クライアント名、商品名、数量、カートン数、カートンサイズ、総重量、到着予定日、トラッキング番号
  - テンプレート設定可能
- 到着確認（M24）でロジセンターが入庫確認 → 在庫数に自動反映
- 入庫通告はAI自動モード対応（§3.11参照）

**② 在庫台帳:**
- クライアント × 商品ごとの在庫数をリアルタイム管理
- 入庫: BAO Flowの案件から自動連携
- 出庫: BAO Flow内の出庫依頼から連携（下記⑤参照）
- 在庫推移履歴（入庫日、出庫日、数量、残高）
- 安全在庫アラート（閾値設定可、在庫がN個以下になったら通知）

**③ 在庫証明書の自動発行:**
- クライアントからの依頼 or 月末自動発行
- PDF形式で出力:
  - 発行日、クライアント名
  - 商品一覧: 商品名、入庫日、現在庫数、保管場所
  - 合計保管数
- クライアントポータルからダウンロード可能

**④ 保管料の自動計算・請求:**
- 保管料の設定（クライアントごと or 全体設定）:
  - 基本保管料: ¥XX/パレット/月 or ¥XX/カートン/月
  - 入庫手数料: ¥XX/回
  - 出庫手数料: ¥XX/回
  - カスタム料金体系にも対応
- 月末に自動で保管料を計算
  - 月中の平均保管量 or 月末時点の保管量（設定可能）
- 保管料請求書を自動生成（§3.4の請求書テンプレートを流用）
- クライアントに自動送信（メール + ポータル表示）

**⑤ 受発注機能（CO-NECT代替 → BAO Flow内蔵）:**

クライアントが保管中の在庫から出庫依頼（＝受注）を行う機能を実装する。

- **クライアント側（クライアントポータル内）:**
  - 現在庫一覧の確認
  - 「出庫依頼」ボタン → 商品・数量・届け先・希望日を入力
  - 出庫依頼の履歴一覧・ステータス確認
  - 複数商品の同時出庫依頼

- **営業・管理者側:**
  - 出庫依頼の受付・承認
  - ロジセンターへの出庫指示メール自動送信（テンプレート設定可）
    - メール内容: 出庫指示ID、クライアント名、商品名、出庫数量、届け先住所、希望出荷日
  - 出庫完了の記録 → 在庫台帳に自動反映
  - 出庫に伴う配送料がある場合の請求管理

- **出庫依頼のステータスフロー:**
  ```
  依頼受付 → 承認 → 出庫指示送信済み → 出荷済み → 配送完了
  ```

- **出庫依頼のデータ:**
  - 出庫依頼ID（自動採番: SO-YYYYMMDD-NNN）
  - クライアント / 商品 / 数量
  - 届け先住所 / 希望出荷日
  - ステータス
  - ロジセンターへの指示メール送信日時
  - 出荷日 / トラッキング番号（あれば）
  - 配送料（あれば）

**⑥ ロジセンター連携画面:**
- ロジセンター担当者向けの簡易画面（将来）
  - 入庫予定一覧
  - 入庫確認ボタン
  - 出庫指示一覧・出庫完了記録
  - 在庫棚卸し入力

**データフロー:**
```
【入庫フロー】
BAO Flow案件（発送済み M22）
    ↓ 自動メール
ロジセンター（入庫通告受領）
    ↓ 到着・入庫確認
BAO Flow在庫台帳（入庫記録・在庫数更新）

【出庫フロー】
クライアント（BAO Flowポータルで出庫依頼）
    ↓
営業（承認）
    ↓ 自動メール
ロジセンター（出庫指示受領 → 出荷）
    ↓ 出荷完了報告
BAO Flow在庫台帳（出庫記録・在庫数更新）

【月次処理】
BAO Flow → 在庫証明書PDF自動発行 → クライアント送信
BAO Flow → 保管料自動計算 → 保管料請求書発行 → クライアント送信
```

### 3.21 AIカタログ取り込み（カタログ自動解析）

工場やAlibabaから受け取ったカタログ（PDF/画像）をアップロードすると、AIが自動で商品情報を抽出し、BAO Flowの商品カタログ（§3.17）に登録する機能。

**処理フロー:**
```
カタログファイルアップロード（PDF/PNG/JPG、複数ページ対応）
    ↓ AIによるOCR + 画像認識
各ページから商品を検出・分割
    ↓ AI解析
商品ごとに仕様情報を抽出:
  - 商品名、サイズ（寸法）、材質
  - 印刷方式、色数
  - 最小ロット数、参考単価
  - 商品画像の切り出し
    ↓ 確認・修正
営業が抽出結果をレビュー・修正（テーブル形式で一覧表示）
    ↓ 一括登録
商品カタログに一括追加
```

**対応する入力フォーマット:**
- 工場のカタログPDF（中国語/英語）
- AlibabaのProductページのスクリーンショット
- Excelの商品リスト
- 個別商品画像（背景自動切り抜き対応）

**AI解析の対象フィールド:**
- 商品名（日本語に自動翻訳）
- サイズ（L×W×H mm）
- 材質・素材
- 印刷方式（オフセット、フレキソ等）
- 参考単価レンジ
- MOQ（最小注文数量）
- 商品画像

**精度向上の仕組み:**
- 営業の修正をフィードバックとしてAIモデルが学習
- 同じ工場のカタログは前回のパターンを記憶
- 認識精度が低い項目は「要確認」マークを付与

**Phase:** Phase 4（AI自動化フェーズ）


### 3.22 工場価格マトリクス & 即時見積もり（Smart Quote）

**目的:** 工場に都度問い合わせなくても、蓄積された価格データから即座に概算見積もりを生成できるようにする。スケール時（月20〜30案件）に見積もりリードタイムを「数日→数分」に短縮する最重要機能。

**① 価格マトリクス（Factory Price Matrix）:**

工場×商品タイプごとに、以下の4軸の組み合わせで価格を管理:

| 軸 | 例 |
|---|---|
| **数量帯** | 1,000 / 3,000 / 5,000 / 10,000 / 20,000 / 30,000 / 50,000 |
| **素材** | PET / PP / 紙（コート紙/クラフト紙/段ボール）/ ブリキ / アルミ / PLA |
| **サイズ** | S / M / L / カスタム（mm指定） |
| **印刷** | 無地 / 1色 / 2色 / フルカラー / 特色(Pantone) |

- 工場ごとに価格マトリクスを登録（手動 or Excelテンプレート一括取込）
- 同一セル（数量×素材×サイズ×印刷）に複数工場の価格が入る → **自動的に相見積もり比較表になる**
- 価格には有効期限を設定（デフォルト: 登録から6ヶ月。工場に定期的に更新依頼）

**② 実データの自動蓄積:**

過去の案件で確定した実際の発注価格を、価格マトリクスに自動フィードバック:
- 案件の発注確定時（M10）→ 工場/商品タイプ/数量/素材/サイズ/印刷/確定単価を自動記録
- 同じ工場×同じ条件で新しい実績があれば、マトリクスの価格を更新提案
- 価格推移のトレンドも記録（為替変動・原材料費変動の影響を可視化）

**③ AI価格推定（Price Estimator）:**

蓄積データが少ない組み合わせでも、AIが推定価格を算出:
- 同じ工場の類似商品の価格から補間
- 数量に対する価格カーブ（通常: 数量2倍で単価15〜25%減）を学習
- 素材・印刷のコスト差係数を過去データから算出
- **信頼度スコア**を表示（実データあり: ★★★、AI推定(類似データ多): ★★☆、AI推定(データ少): ★☆☆）
- ★☆☆の場合は「工場に確認を推奨」と表示

**④ 即時見積もりフロー（Smart Quote Flow）:**

1. クライアントから「PETカップ 8oz 10,000個 フルカラー印刷」の依頼が来る
2. BAO Flowが価格マトリクスを参照 → 対応可能な工場を自動リストアップ
3. 工場別の原価比較表を自動生成（単価/送料推定/合計/リードタイム/信頼度）
4. 営業が最適な工場を選択 → 掛率適用 → **見積書ドラフトを数分で生成**
5. 信頼度が低い場合 → 工場に正式見積もり依頼（従来フロー）を並行実行

**⑤ 工場への価格表テンプレート配布:**

BAO Flowから工場に送る**標準価格表テンプレート（Excel）**を生成:
- 商品カテゴリ別のマトリクス（数量帯×サイズが行、素材×印刷が列）
- 工場が価格を埋めて返送 → AIが自動パースして価格マトリクスに取込
- 定期更新リマインダー（6ヶ月ごとに「価格表の更新をお願いします」と自動送信）

**Phase:** Phase 3（工場ポータルと同時期。ただし②の自動蓄積はPhase 1から裏で開始）

### 3.23 食品衛生法 品目登録管理 & 規格検査管理

**目的:** 食品に触れる器具・容器包装の輸入時に必要な食品衛生法の規格検査・品目登録を効率的に管理。営業が「この商品は検査済みか？」を即座に判断でき、必要な場合は委託先に検査手配を依頼するだけで済む状態を作る。

**現行の検査フロー（実績に基づく）:**
1. 工場がサンプルを中国の検査機関に送付（先行サンプル検査）
2. 中国の外国公的検査機関（例: IQTC 広州海関技術中心）が厚労省告示370号基準で規格検査を実施
3. 試験成績書（検測報告 / Test Report）を取得（約10営業日）
4. 試験成績書を食品等輸入届出書に添付して検疫所に提出 → 届出済証を取得
5. 品目登録を行えば、以降の同一商品は検査不要

**検査の委託先:** イーウー（YIWU PASSPORT）/ 海源 / D2D（一括依頼時）
※ 検査手配自体は委託先が行い、BAO Flowは成績書の管理・照合に専念

**検査が必要な主な材質（御社商品に関連するもの）:**
- **合成樹脂（PET/PP/PS等）:** 一般規格4項目（Cd、Pb、重金属、KMnO₄消費量）+ 材質別個別規格（PETの場合: Sb・Ge溶出量）
- **金属缶（ブリキ等）:** As、Pb、Cd等の溶出試験
- **紙製品:** 食品に直接接触する場合のみ検査対象（外装箱は不要）
- **ガラス・陶磁器:** Pb、Cdの溶出試験

**品目登録制度:**
- 初回: 規格検査合格 → 品目登録要請書を検疫所に提出 → 品目登録番号を取得
- 2回目以降: **同一製造者・同一材質・同一形状なら再検査不要**（器具・容器包装は有効期限なし = 永久有効）
- ただし色違い・形状違いは別検体扱いの場合あり（検疫所判断）

**BAO Flowでの管理:**

- **試験成績書台帳:** 報告番号（例: 0726000357）/ 検査機関（例: IQTC）/ 申請人 / 商品名 / 材質 / 製造者 / 検査日 / 結論（合格/不合格）/ 試験成績書PDF
- **品目登録台帳:** 品目登録番号 / 紐づく試験成績書 / 材質 / 製造者（工場）/ 形状 / 検疫所
- **案件との自動照合:** 新規案件の商品が「登録済みの工場×材質×形状」と一致 → 「✅ 品目登録済み（検査不要）」と自動表示
- **未登録アラート:** 新しい工場×材質の組み合わせ → 「⚠️ 要検査 — 委託先に検査手配を依頼してください」+ 推定検査費用を原価に自動加算
- **検査費用の自動計算:** 材質に応じたデフォルト検査費用を原価計算に組み込み（初回のみ。品目登録後はゼロ）
- **登録推奨レコメンド:** 同じ工場×材質の案件が2件目になった時点で「品目登録を推奨」と通知

**Phase:** Phase 1（原価計算に組み込み。台帳管理は簡易版から開始）

### 3.27 輸入手続き管理（Import Flow Manager）

**目的:** YIWU PASSPORT月額¥30,000を廃止し、輸入手続きをBAO Flow主導で一元管理する。営業が「製造完了→日本到着」までの全工程を最小限の手間で遂行できる統合ワークフロー。

**基本方針:**
- 「書類・申請はBAO Flowで自動化、物理的な輸送と検査は外注」
- 営業の手間を最小化することが最優先。コスト最適化は重要だが、複雑さとのバランスを取る
- 自社でできる部分（FAINS届出、書類準備）は自社で、複雑な部分（通関・検査手配）はD2Dに一括委託
- **食品届出はFAINS自社申請を基本方針とする**（BAO Flow届出書自動生成 → FAINSにコピペ。品目登録済み: 10分/件・¥0、未登録: 20-30分/件・¥0）

## 4. 非機能要件

### 4.1 パフォーマンス
- ページロード: 3秒以内
- 原価計算: リアルタイム（入力変更時に即時反映）
- チャットメッセージ: リアルタイム配信
- 翻訳: 送信時に自動翻訳（2秒以内）

### 4.2 セキュリティ
- ロールベースのアクセス制御（Admin / 営業 / クライアント / 工場）
- クライアントには原価・掛率・工場情報を一切非表示
- 工場にはクライアント情報・販売価格を非表示
- 工場間の情報を相互非表示（相見積もりの場合、他社見積もりは見えない）

### 4.3 多言語対応
- 管理画面: 日本語
- クライアントポータル: 日本語
- 工場向け画面: 中国語（簡体字）+ 英語（切り替え可能）
- チャット: 日中同時翻訳

### 4.4 レスポンシブ対応
- PC / タブレット / スマートフォン対応

### 4.5 通知
- クライアント: メール通知（ステータス変更 + チャットメッセージ） + システム内チャット
- 営業: ダッシュボード通知 + システム内チャット + 日次サマリーメール
- 工場: メール通知（ステータス変更 + チャットメッセージ） + システム内チャット
- 将来: LINE / WeChat / Lark連携の可能性

## 5. 技術スタック（推奨）

| レイヤー | 技術 | 理由 |
|----------|------|------|
| **フロントエンド** | Next.js (React) | Vercelでのデプロイ経験あり |
| **バックエンド** | Next.js API Routes or 別サーバー | フロントと統合可能 |
| **データベース** | Supabase (PostgreSQL) | 使用経験あり、RLS対応 |
| **認証** | Supabase Auth | ロールベース認証が容易 |
| **ファイルストレージ** | Supabase Storage | PDF・画像・Excel・デザインデータの保管 |
| **ホスティング** | Vercel（フロント）+ Railway or Supabase（バックエンド） | 使用経験あり |
| **為替API** | ExchangeRate-API or Open Exchange Rates | リアルタイム為替取得 |
| **PDF生成** | @react-pdf/renderer or puppeteer | 帳票・在庫証明書の自動生成 |
| **リアルタイムチャット** | Supabase Realtime | 案件別チャット + 通知 |
| **メールブリッジ** | Resend or SendGrid + Inbound Email Parsing | チャット↔メール双方向連携 |
| **翻訳** | DeepL API or Google Cloud Translation | 日中翻訳の精度はDeepLが優秀 |
| **AI/LLM** | Claude API (Anthropic) | Excel自動パース、AIアシスタント、要望解析 |
| **Excel解析** | SheetJS + Claude API | 工場Excelの自動読み取り |
| **配送トラッキング** | 17track API or Ship24 API | 国際配送の追跡 |

### 6.2 RLS（行レベルセキュリティ）ポリシー

- **クライアント**: 自社の案件・見積もり（販売価格のみ）・注文履歴・在庫情報・チャット（クライアント-営業）のみ閲覧可能。原価・掛率・工場情報は完全非表示
- **工場担当者**: 自分が担当する案件の仕様・チャット（営業-工場）のみ閲覧可能。クライアント情報・販売価格は完全非表示。他工場の見積もりも非表示
- **営業担当者**: 自分が担当する案件の全情報を閲覧・編集可能
- **管理者**: 全データに対するフルアクセス

## 9. 原価計算の検証例

### 例: 株式会社セガワ - Peanuts Craft bag（20,000個）

| 項目 | Lark計算値 | 実際の請求書（Misoca） |
|------|-----------|---------------------|
| 数量 | 20,000 | 20,000 |
| 工場単価 | $0.049 | — |
| 送料 | $127 | — |
| 版代 | $0 | — |
| その他費用 | $0 | — |
| **ドル合計** | **$1,107** | — |
| 商品単価（ドル） | $0.055 | — |
| 掛率 | 0.65 | — |
| 販売単価（ドル） | $0.085→$0.09 | — |
| 為替レート | ¥157 | — |
| **販売単価（円）** | **¥13** | **¥14** |
| **請求合計** | **¥260,000** | **¥280,000** |
| **請求合計（税込）** | **¥286,000** | — |
| サンプル費用 | — | ¥30,030（60,060円の弊社半額負担） |
| **合計請求額（税込）** | — | **¥341,033** |

**差異分析:**
- 販売単価¥13 vs ¥14: 掛率の微調整または端数の切り上げ方の違い
- サンプル費用: 60,060円（送料込み）の半額=30,030円をクライアントに請求
- 消費税: (280,000 + 30,030) × 1.1 = 341,033

**→ BAO Flowでは掛率のスライダー調整で販売単価を営業が自由に設定可能にし、この柔軟性を担保する**
